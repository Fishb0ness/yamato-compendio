## Compact Rules

### issue-creation
- Use templates for bug/feature issues; blank issues are disabled.
- Add `status:needs-review` on creation; a maintainer must add `status:approved` before PR.
- Require pre-flight checks (duplicate search, required fields) in every issue.
- For CLI automation, use `gh issue create --template` with populated fields.

### branch-pr
- Enforce issue-first workflow: PRs must reference an approved issue.
- Draft PR bodies with summary, changed files overview, and testing instructions.
- Include changelog-style commit messages (conventional commits recommended).
- Do not open PRs before `status:approved` label exists on the issue.

### skill-creator
- Follow Agent Skills frontmatter format: name, description, metadata.
- Include compact rules section (5-15 lines) with actionable constraints.
- Keep examples short; do not include secrets or large code dumps.
- New skills should include triggers and critical rules for delegators.

### go-testing
- Use `go test` for unit tests; prefer table-driven tests for logic.
- For Bubbletea TUI, rely on provided testing helpers and deterministic inputs.
- Keep tests fast and isolated; avoid network calls in unit tests.
- Use `go test -cover` to generate coverage; integrate with CI if available.

### judgment-day
- Run two blind independent reviews against the same target and synthesize results.
- Iterate fixes and re-judge until both pass or escalate after 2 iterations.
- Provide clear PASS/FAIL reasons, reproduction steps, and suggested fixes.
- Keep reviewer prompts identical to ensure blind comparison.
